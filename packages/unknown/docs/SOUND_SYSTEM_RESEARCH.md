# Звуковая система при большом числе источников — исследование

> Deep-research отчёт (2026-06-06): как игры с десятками одновременных звуков
> (выстрелы, попадания, взрывы, моторы) избегают какофонии, и какие техники
> переносимы на Web Audio API в нашей игре (`SoundManager` / `WebAudioTrack` /
> ECS Sound-сущности). Все утверждения прошли адверсариальную верификацию
> (3 голоса на утверждение); confidence указан по каждому пункту.

## TL;DR

Все большие движки (Wwise, FMOD, Unity, Unreal) сходятся к одному ядру:
**большой пул дешёвых «виртуальных» голосов + маленький бюджет «реальных»**,
которые действительно микшируются. Поверх этого: **лимиты инстансов по типу
звука**, **culling по слышимости** (дистанция + порог громкости), **приоритеты
поверх громкости**, **ducking/sidechain** и **round-robin вариативность
сэмплов**. Web Audio ничего из этого не даёт из коробки — но все строительные
блоки (GainNode-шины, мастер-гейн, пулинг) есть, паттерны переносятся напрямую
в JS-код.

Лозунг Frostbite HDR Audio: **«играй правильные звуки, а не все звуки»**.

## Подтверждённые механизмы (high confidence)

### 1. Двухуровневые голоса: virtual / real

Доминирующий кросс-движковый паттерн. «Виртуальный» голос — только позиция и
громкость, ни декодирования, ни фильтров, почти нулевой CPU. «Реальный» —
полноценно микшируется. Звуки динамически перетекают между уровнями по
слышимости; стоимость кадра ограничена числом реальных голосов.

- FMOD: сотни–тысячи виртуальных, **32–128 реальных** каналов (типично 512–1024 / 32–64)
- Unity: по умолчанию **32 реальных** голоса
- Wwise: ~128 физических на PC

Источники: [Wwise: optimizing voices](https://www.audiokinetic.com/en/blog/how-to-get-a-hold-on-your-voices-optimizing-for-cpu-part-1/), [FMOD virtual voices](https://documentation.help/fmod-studio-api/virtualvoices.html)

### 2. Culling по слышимости: дистанция + порог + поведение

Одной дистанционной аттенюации мало. Wwise-правило: **«Kill if finite else
virtual»** — когда звук падает ниже порога слышимости, *конечный* (one-shot)
звук убивается насовсем, *зацикленный* (мотор) — виртуализируется и
возвращается, когда снова станет слышен. Через этот механизм отсекается
основная масса звуков в игре.

### 3. Лимиты инстансов по типу звука (наш главный инструмент)

Кап на число одновременных инстансов одного ТИПА звука, даже если все слышимы.
В Wwise — на трёх уровнях (звук / шина / глобально, по умолчанию 256), новый
звук стартует только если есть слот на каждом уровне. Это и есть «coalescing»:
20 одновременных выстрелов не складываются в кашу — играют первые N.

При превышении лимита — **resolution rule** (Unreal Sound Concurrency,
default Max Count = 16): Prevent New / Stop Oldest / Stop Farthest /
Stop Lowest Priority / Stop Quietest. Unity: сначала явный приоритет,
потом громкость как tiebreaker.

Источник: [Unreal Sound Concurrency](https://dev.epicgames.com/documentation/en-us/unreal-engine/sound-concurrency-reference-guide)

### 4. Приоритет сильнее громкости

Явный приоритет голоса всегда побеждает дистанцию/громкость: голос с высоким
приоритетом никогда не вытесняется низкоприоритетным (FMOD). Best practice —
**мало грубых уровней** (FMOD: Lowest/Low/Medium/High/Highest), а не тонкая
шкала.

### 5. HDR audio = culling, а не компрессия

Frostbite (Battlefield: Bad Company): измерить громкость в точке слушателя,
отмасштабировать все источники, играть только важное. **Приоритет выводится из
громкости**, потому что громкость — хорошее приближение важности в экшене.
«130 dB динамического диапазона» — иллюзия, созданная culling'ом по громкости,
не буквальный выход. Работает на логических значениях громкости, не трогая
сэмплы.

Источники: [Frostbite: Adaptive mixing (GDC)](https://www.slideshare.net/slideshow/adaptive-mixing-in-frostbite/3128152), [EA: HDR audio in BFBC](https://www.ea.com/frostbite/news/how-hdr-audio-makes-battlefield-bad-company-go-boom)

### 6. Sidechain / ducking

Уровень важного сигнала (взрыв) приглушает менее важный (моторы). Динамический
sidechain (следящий за огибающей транзиентов) звучит естественнее, чем
event-based ducking с фиксированной длительностью. В Wwise: Meter-эффект на
шине взрывов → RTPC → аттенюация шины моторов.

Источник: [Side-chaining in Wwise](https://mcvuk.com/development-news/audio-guide-side-chaining-in-wwise/)

### 7. Round-robin вариативность

Пул похожих-но-разных сэмплов на каждый тип one-shot'а убивает
«пулемётный» артефакт повторения идентичного сэмпла. Обязательно для
выстрелов/попаданий, которых десятки в секунду. Дёшево: несколько вариантов +
случайный детюн/гейн (±5%).

### ⚠️ Опровергнуто верификацией

Утверждение «DynamicsCompressorNode поднимает тихие части микса» — **ложно**
(голоса 1-2). Компрессор только ослабляет громкое (+makeup gain), тихое не
усиливает. Мастер-компрессор корректно описывать только как **safety limiter
от клиппинга**, а не замену динамическому миксу.

## Архитектура для нашей игры (Web Audio + ECS)

> Синтез подтверждённых паттернов (medium confidence как целое — каждая
> составляющая high). Привязка к текущему коду: `SoundManager.ts`,
> `WebAudioTrack.ts`, `Entities/Sound.ts` (`spawnSoundAtPosition`).

### Граф шин

```
shot ──┐
hit ───┤
boom ──┼── busGain (по типу) ──► masterGain (<1) ──► DynamicsCompressor ──► destination
engine─┘                                              (safety limiter)
```

GainNode на каждый тип (`SoundType`): выстрелы, попадания, взрывы, моторы.
Мастер-гейн < 1, чтобы сумма не клипповала; компрессор — только страховка.

### Voice manager (JS, поверх SoundManager)

- **Бюджет реальных голосов**: стартовать с ~24–32 (число из нативных движков,
  на Web Audio надо мерить — см. открытые вопросы).
- **Слышимость** = базовая громкость типа × аттенюация по дистанции до камеры
  (в top-down истинного 3D нет — дистанция до центра экрана / слушателя).
- **Порог слышимости**: ниже порога — one-shot не стартует вовсе («kill if
  finite»), зацикленный мотор виртуализируется (gain → 0, позиция продолжает
  трекаться ECS-сущностью; ре-материализация при возвращении в слышимость).
- **Вытеснение** при переполнении бюджета: сначала низший приоритет, затем
  самый дальний, затем самый старый. Приоритеты грубые:
  `Explosion > Shoot/Hit > Engine > Ambient`.

### Лимиты инстансов по типу (главное против какофонии)

| Тип | Кап | Resolution rule |
|---|---|---|
| TankShoot | 4–6 | Prevent New (новый выстрел в залпе ничего не добавляет) |
| Hit | 3–4 | Stop Oldest |
| Explosion | 2–3 | Stop Quietest |
| Engine (loop) | 4–8 | Stop Farthest (дальние виртуализировать) |

Залп из 20 выстрелов = 4–6 реально играющих + round-robin вариация — звучит
как залп, а не как клиппинг.

### Round-robin

2–4 варианта сэмпла на тип + случайный `playbackRate` ±5% и гейн ±2 dB на
каждый старт. Это дешевле и важнее, чем поднимать капы.

### Ducking

Взрыв стартует → запланированная огибающая на `engineBus.gain`
(`setTargetAtTime`: ↓ до ~0.4 за 30 мс, возврат за ~400 мс). Event-based
ducking достаточен для нашего масштаба; honest sidechain (AnalyserNode/
AudioWorklet-метр на шине взрывов) — только если event-based будет звучать
механически.

### Порядок внедрения

1. Шины + мастер-гейн + лимитер (день работы, сразу убирает клиппинг)
2. Капы инстансов по типу + resolution rules (главный анти-какофония шаг)
3. Round-robin + рандомизация pitch/gain
4. Бюджет голосов + culling по слышимости (когда сцен станет больше)
5. Ducking взрывами моторов
6. HDR-style loudness priority — только если 1–5 не хватит (вероятно, хватит)

## Открытые вопросы (не закрыты источниками)

- Реальный потолок одновременных `AudioBufferSourceNode` на среднем железе в
  браузере (числа 32–64 взяты из нативных движков; один аудио-поток в браузере
  может опустить потолок) — мерить на своём контенте.
- Как дешевле виртуализировать loop в Web Audio: gain → 0 с живым source-нодом
  или stop + пересоздание с трекингом elapsed time (как «play from elapsed
  time» в FMOD/Wwise).
- Какая метрика «громкости-важности» лучше для top-down: квадрат дистанции до
  камеры, дизайнерская базовая громкость типа, или гибрид (HDR-style).
